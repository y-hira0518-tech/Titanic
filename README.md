# Titanic - Survival Prediction

Kaggleの Titanic - Machine Learning from Disaster を題材に、前処理・特徴量エンジニアリング・モデル比較・ハイパーパラメータチューニング・モデル解釈までを行った分析です。

## 環境

- Google Colab
- Kaggleデータ取得は `kaggle.json` を `files.upload()` でアップロードする方式

## 前処理・特徴量エンジニアリング

交差検証時のデータリーケージ(検証データの情報が学習データの統計量計算に混入すること)を避けるため、`scikit-learn` の `Pipeline` に組み込んだカスタム `Transformer` として実装しています。

- **Title(敬称)抽出**: `Name` から `Mr`/`Mrs`/`Miss`/`Master`/`Rare` 等の敬称を抽出。単なる性別より粒度の細かい年齢層・婚姻状況の情報を持つことをデータで確認した上で採用(Master中央値3.5歳 vs Mr中央値30歳など、Pclass・Sexの分割だけでは失われる差を捉えられることを検証済み)
- **Age(年齢)補完**: Title別の中央値で補完。中央値は交差検証の学習fold内でのみ算出し、検証foldには適用のみを行うことでリークを防止
- **Fare(運賃)補完**: Pclass別の中央値で補完(同じくfit時=学習foldのみで算出)
- **Embarked(乗船港)補完**: 学習foldの最頻値で補完
- **HasCabin**: Cabin(客室番号)の記録有無を二値特徴量化。Cabin自体は欠損率が約77%と高く、デッキ別(先頭文字)の生存率差はサンプル数が小さいカテゴリ(1件のみのケースを含む)を含み交差検証との相性が悪いと判断し、有無のみを採用
- **FamilySize / IsAlone**: `SibSp + Parch + 1` で家族人数を算出し、独り身かどうかを二値特徴量化
- **不採用とした特徴量**: チケット共有グループサイズ、およびそれに依存するFare Per Personは、交差検証の各foldで値が変動し正確な算出ができないため不採用

## モデル比較

線形モデル・距離ベース・カーネルベース・単一木・バギング・ブースティングと手法の系統を広くカバーする6モデルを比較しました。

- ロジスティック回帰
- K近傍法(KNN)
- サポートベクターマシン(SVM)
- 決定木
- ランダムフォレスト
- LightGBM

各モデルはOptunaで1モデルあたり最大120秒のタイムアウトを設定してハイパーパラメータチューニングを実施(探索用CVは3-fold、最終評価は5-fold)。SVMは計算コストが高いため、探索中は `probability=False` とし `scoring="roc_auc"` が内部的に `decision_function` にフォールバックする形で評価、`cache_size=1000`・`n_jobs=-1` で高速化しています。

| モデル | ROC-AUC(5-fold平均) |
|---|---|
| LightGBM | 0.8902 |
| Random Forest | 0.8863 |
| ロジスティック回帰 | 0.8743 |
| KNN | 0.8741 |
| SVM | 0.8637 |
| 決定木 | 0.8626 |

ツリー系アンサンブル(LightGBM・Random Forest)が優位という結果になりました。

## モデル解釈(SHAP)

最良モデル(LightGBM)についてSHAP(SHapley Additive exPlanations)分析を実施し、平均絶対SHAP値による特徴量重要度と、LightGBM組み込みのGain基準の重要度を算出・比較しました。

**上位の特徴量(平均絶対SHAP値)**

1. Title = Mr(成人男性)
2. Sex = female
3. Pclass = 3
4. Fare
5. Age
6. HasCabin
7. Sex = male
8. FamilySize

Title(Mr)が両指標で最重要特徴量となっており、Titanicでよく知られる「女性・子供優先」という生存傾向の構造をモデルが捉えていることが確認できました。また、FareはGain基準では2位である一方SHAP基準では4位となっており、「学習時の分岐のしやすさ」と「実際の予測値への寄与度」が必ずしも一致しない例として観察されました。

## 開発方針

- コードはPipeline化し、交差検証時のデータリーケージを構造的に防止する
- 特徴量エンジニアリングは仮定で採用せず、実データでの検証(グループ別統計量の比較、既存特徴量との重複確認)を経てから採用可否を判断する
- 特徴量重要度の可視化と、エンジニアリングした特徴量の効果分析を必ず行う
